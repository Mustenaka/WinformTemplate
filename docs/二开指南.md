# 二开指南：新增一个业务模块

本文按 `docs/REFACTOR_PLAN.md` §9 的 6 步编写，以当前 Product 模块为范例。目标是在约 30 分钟内加出一个可运行的新业务模块，并保持三种数据源实现一致。

示例里用 `Order` 作为新模块名。实际开发时替换为你的业务名即可。

## 0. 开始前确认

先确认项目仍然满足这些规则：

- Service / ViewModel 不引用 `DbContext`、`HttpClient`、EF 类型或 `Expression<Func<...>>`。
- 业务层只依赖 `IXxxRepository` 抽象。
- 查询条件使用 `QueryRequest`，分页、搜索、排序和过滤由仓储下推。
- WebApi 后端未实现时，传输失败抛 `DataSourceUnavailableException`，不要返回空集合伪装成功。
- EF 模块使用独立 SQLite 文件；新增 EF 模块后要有真实启动序列测试覆盖。

## 1. Model

目录建议：

```text
WinformTemplate/Src/Business/Order/Model/
```

参考 Product：

```text
WinformTemplate/Src/Business/Template/Model/ProductModel.cs
```

要求：

- 使用 EF attribute 标明表名和列名。
- 主键使用 `long Id` 或本模块统一命名。
- UI 展示用字段可加 `[NotMapped]`，不要让 ViewModel 拼装重复格式化逻辑。
- 状态字段语义要写清楚。例如 Sys 中 `false` 表示有效。

## 2. Repository 契约

目录建议：

```text
WinformTemplate/Src/Business/Order/Repositories/
```

定义模块专属仓储接口：

```csharp
public interface IOrderRepository : IRepository<OrderModel>
{
    Task<PagedResult<OrderModel>> SearchOrdersAsync(
        string? keyword = null,
        int? status = null,
        DateTime? startDate = null,
        DateTime? endDate = null,
        int pageIndex = 1,
        int pageSize = 20,
        string? orderBy = null,
        bool ascending = true);
}
```

契约原则：

- 继承 `IRepository<T>`。
- 复杂查询提供具名方法，例如 Product 的 `SearchProductsAsync`、`IsCodeExistsAsync`。
- 不把 `Expression<Func<...>>` 暴露到接口，因为 WebApi 无法传递表达式。
- `QueryAsync(QueryRequest)` 是跨源统一入口，具名方法可以由 `QueryAsync` 映射，或反过来由 Service 构造 `QueryRequest` 调用。

## 3. 三种 Repository 实现

每个模块都提供三套实现：

```text
Repositories/
  EfCore/EfOrderRepository.cs
  WebApi/ApiOrderRepository.cs
  Local/LocalOrderRepository.cs
```

### Ef 实现

参考：

```text
WinformTemplate/Src/Business/Template/Repositories/EfCore/EfProductRepository.cs
```

要求：

- 继承 `EfRepositoryBase<OrderModel>`。
- 查询在数据库端完成：`Where`、`OrderBy`、`Skip`、`Take` 都在 IQueryable 上执行。
- 排序字段要做白名单，非法 `SortBy` 安全忽略或使用默认排序。
- 写操作返回真实结果，未命中返回 `false`，不可达异常继续抛出。

### WebApi 实现

参考：

```text
WinformTemplate/Src/Business/Template/Repositories/WebApi/ApiProductRepository.cs
docs/api-contract.md
```

要求：

- 继承 `ApiRepositoryBase<OrderModel>`。
- URL、分页参数、排序参数、响应包络必须与 `docs/api-contract.md` 一致。
- 业务未命中按契约返回 `null`、空集合、`false` 或 `0`。
- 传输失败由基类转成 `DataSourceUnavailableException`，由 ViewModel 显示“未连接后端”。

### Local 实现

参考：

```text
WinformTemplate/Src/Business/Template/Repositories/Local/LocalProductRepository.cs
WinformTemplate/Resources/MockData/products.json
```

要求：

- 继承 `LocalRepositoryBase<OrderModel>`。
- 在 `Resources/MockData/orders.json` 提供样例数据。
- CRUD 只在内存集合生效，默认不回写磁盘。这个设计是为了演示数据可重置。
- 若测试会修改 Local 数据，使用独立临时种子目录，避免静态 per-path 缓存污染用例。

## 4. Service

目录建议：

```text
WinformTemplate/Src/Business/Order/Service/
```

参考：

```text
WinformTemplate/Src/Business/Template/Service/ProductService.cs
```

Service 只依赖仓储契约：

```csharp
public sealed class OrderService : IOrderService
{
    private readonly IOrderRepository _orderRepository;

    public OrderService(IOrderRepository orderRepository)
    {
        _orderRepository = orderRepository;
    }

    public async Task<(List<OrderModel> Items, int TotalCount)> SearchOrdersAsync(
        string? keyword,
        int pageIndex,
        int pageSize,
        string? orderBy,
        bool ascending)
    {
        var result = await _orderRepository.QueryAsync(new QueryRequest
        {
            Page = Math.Max(pageIndex, 1),
            PageSize = Math.Max(pageSize, 1),
            Keyword = keyword,
            SortBy = orderBy,
            Desc = !ascending
        });

        return (result.Items.ToList(), result.Total);
    }
}
```

Checklist：

- 不引用 EF、DbContext、HttpClient。
- 不用 `GetAll().Where(...)` 做内存过滤。
- 对业务校验返回明确消息。
- 对数据源不可达不要吞掉异常后返回空集合；需要 UI 降级提示时让 `DataSourceUnavailableException` 进入 ViewModel。

## 5. ViewModel

目录建议：

```text
WinformTemplate/Src/Business/Order/ViewModel/
```

参考：

```text
WinformTemplate/Src/Business/Template/ViewModel/ProductManagementViewModel.cs
```

ViewModel 负责：

- 当前页、页大小、总数、总页数。
- 搜索关键字、筛选条件、排序字段。
- `LoadDataAsync`、`SearchAsync`、`GoToPageAsync`、`ApplySortAsync`。
- CRUD 成功后刷新当前页。
- 导出等非 UI 业务逻辑。

测试要求：

- 覆盖分页参数是否传入 Service。
- 覆盖新增、修改、删除后的刷新。
- 覆盖导出或关键业务动作。

## 6. UserControl + 注册 + 菜单

### UI

目录建议：

```text
WinformTemplate/UI/Business/Order/
```

参考：

```text
WinformTemplate/UI/Business/Template/Product/ProductManagementControl.cs
```

要求：

- UI 是薄壳，业务逻辑放 ViewModel。
- AntdUI Table 显示列表。
- 分页、搜索、排序控件调用 ViewModel 方法。
- 增删改使用对话框或独立编辑面板。
- 避免在 UI 内直接访问仓储、DbContext、HttpClient。

### 注册仓储与服务

在 `Src/Bootstrap/AppServiceRegistration.cs` 添加：

```csharp
services.AddModuleRepository<IOrderRepository, EfOrderRepository, ApiOrderRepository, LocalOrderRepository>("Order");
services.AddScoped<IOrderService, OrderService>();
services.AddTransient<OrderManagementViewModel>();
services.AddTransient<OrderManagementControl>();
```

如果新增 EF 模块，还要注册该模块 DbContext 和 `IDatabaseInitializer`。默认 SQLite 下模块会解析为独立文件，例如 `order.db`。

### 注册页面

在 `Src/Navigation/PageRegistryDefaultPages.cs` 添加：

```csharp
pageRegistry.Register("/order/list", sp => sp.GetRequiredService<OrderManagementControl>());
```

### 菜单种子

必须同时改两处：

- EF 种子：例如 `SysDatabaseInitializer` 里的 `SysMenuModel` 和 `SysRoleAuthModel`。
- Local MockData：`Resources/MockData/sysMenus.json` 和 `sysRoleAuths.json`。

新增菜单 key 推荐使用稳定 URL：

```text
/order/list
```

不要 EF 用一个 key、Local 用另一个 key、页面注册表再用第三个 key。

## 必跑测试

新增模块后至少跑：

```powershell
dotnet build WinformTemplate.sln -warnaserror
dotnet test WinformTemplate.sln
```

建议补这些测试：

- Ef + Local 仓储契约一致测试。
- ViewModel 分页/CRUD 测试。
- WebApi mock handler 映射和不可达降级测试。
- 如果改了菜单，确认 `SysMenuSeedUrls_AreConsistentAcrossEfLocalAndRegisteredPages` 绿。
- 如果新增 EF 模块，补类似 `AppStartupIntegrationTests` 的真实启动序列覆盖，确认默认 SQLite 能建库、种子和查询。

## 两个踩坑 checklist

1. 菜单和页面 key 漂移

   改菜单或页面注册后，必须确认三处一致：

   - `SysDatabaseInitializer`
   - `Resources/MockData/sysMenus.json`
   - `PageRegistryDefaultPages.RegisterDefaultPages()`

   守卫测试 `SysMenuSeedUrls_AreConsistentAcrossEfLocalAndRegisteredPages` 必须通过。

2. 多 DbContext 共用 SQLite 文件

   不要让多个 EF 模块共用同一个 SQLite 文件再调用 `EnsureCreated`。EF Core 不会为后执行的 Context 创建表。默认方案是每个模块一个文件：`sys.db`、`template.db`、未来的 `order.db`。

   新增 EF 模块后，真实启动序列测试必须覆盖它，避免单独的 DbContext 单测掩盖 App 启动崩溃。
